# /blueprint

Parse $ARGUMENTS:

| Pattern | Action |
|---------|--------|
| `setup` | Check if `lib/hook-installer.cjs` exists in the project root. If not, report: "Blueprint installers not found in this project. The Blueprint `lib/` installer scripts are not bundled with the marketplace plugin; this subcommand only runs in a project that already has the `lib/` and `components/` directories at its root. Skipping setup â€” every other 4ge command works without Blueprint." If found: ask the user for project_name, tech_stack, tier (`lite`/`standard`/`full`), and setup_mode. The config schema (`components/configs/.blueprint-config.json.schema`) requires four fields â€” `project_name`, `os`, `tier`, `blueprint_date` â€” and `generate.cjs` now validates fail-closed (an invalid or under-specified config aborts before any file is touched). So also set `os` (detect the platform: `windows`, `mac`, or `linux`) and `blueprint_date` (today's date, `YYYY-MM-DD`). Validate against the schema. Write answers to `.blueprint-config.json` with `"project_path": "."` (the positional root passed to `generate.cjs` must resolve to the same directory as `project_path`, or the run aborts with a target-root mismatch). Then run `node lib/generate.cjs . .blueprint-config.json` from the project root â€” this is the 13-step orchestrator (`lib/generate.cjs`) that calls all four installers plus the OS substrate installer, writes `CLAUDE.md`, `.4ge/config.json`, and `.blueprint-manifest.json`, and updates `.gitignore`. Do **not** invoke the individual installers (`hook-installer.cjs` etc.) directly or via `node -e "require(...)"` â€” none of them writes `.blueprint-manifest.json`, so `/blueprint status` would never find an installation, and a bare `node -e "require(...)"` never runs the installer's `require.main === module` block at all (silent no-op, exit 0). |
| `update` | Re-run Blueprint installation using existing `.blueprint-config.json`. Check `lib/hook-installer.cjs` exists first (report same message as setup if missing). The same fail-closed validation applies: if the existing config is missing any of `project_name`/`os`/`tier`/`blueprint_date`, or its `project_path` does not resolve to the project root (use `"."`), `generate.cjs` aborts before touching disk â€” add the missing fields or fix `project_path` first. Run `node lib/generate.cjs . .blueprint-config.json` (same orchestrator as setup). By default this is idempotent â€” hooks/agents/rules that already exist are kept (project-local edits preserved), only newly-added catalog entries and skills (which always refresh from source) are written; managed files skipped this run are carried forward in the manifest, never dropped. To force a full re-apply that overwrites existing files (every file/skill directory is backed up **and verified** before any original is deleted, and the whole run auto-rolls-back on failure), add `--force`: `node lib/generate.cjs . .blueprint-config.json --force`. If `.blueprint-config.json` does not exist, report "No config found â€” run `/blueprint setup` first." |
| `status` | Read `.blueprint-manifest.json` and display installation health: version, file count, installed date. If missing, report "No Blueprint installation found." |
| (empty) | Show available subcommands: setup, update, status. |
